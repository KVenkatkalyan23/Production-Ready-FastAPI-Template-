Add Success/Error Response Models
Objective
Create reusable response schemas so all endpoints return a consistent success/error structure.
________________


Scope
Includes:
* success response schema
* error response schema
* shared base response schema
* support for message, data, error
Excludes:
* exception handlers
* business logic
* logging
* auth-specific response models
________________


Files
* app/schemas/base.py
* app/schemas/success.py
* app/schemas/error.py
Optional:
* app/schemas/__init__.py
________________


Target Format
Success
{
 "success": true,
 "message": "Request completed successfully",
 "data": {}
}
Error
{
 "success": false,
 "message": "Request failed",
 "error": {
   "code": "...",
   "details": {}
 }
}
________________


What to implement
base.py
Create BaseResponse with:
* success
* message
success.py
Create SuccessResponse extending BaseResponse with:
* success: bool = True
* data: Any | None = None
error.py
Create:
* ErrorDetail
   * code: str
   * details: Any | None = None
* ErrorResponse extending BaseResponse with:
   * success: bool = False
   * error: ErrorDetail
________________


Requirements
* keep schemas generic
* no auth/user/DB-specific fields
* no HTTP status code inside schema
* OpenAPI-friendly
* reusable in routes and handlers
________________


Acceptance
* BaseResponse exists
* SuccessResponse exists
* ErrorDetail exists
* ErrorResponse exists
* all success responses follow one structure
* all error responses follow one structure
________________


Validation
Success
SuccessResponse(message="OK", data={"status": "healthy"})
Error
ErrorResponse(
   message="Validation failed",
   error=ErrorDetail(code="VALIDATION_ERROR", details={"email": "invalid"})
)
________________


Deliverables
* reusable base response schema
* reusable success response schema
* reusable error response schema
* ready for future route and exception-handler integration
